# Codex 接手上下文

这个文档给下一次启动的 Codex 使用，用来快速理解项目目标、当前状态、工作原则和已知问题。

如果重新启动 Codex，建议先阅读这个文档，再阅读 `PROGRESS_MAP.md`、`TODO_AND_BUGS.md` 和 `PROJECT_STATE.md`。

## 项目目标

这是一个个人管理面板，用来管理任务、倒计时事件、定时检查、电费监控、提醒、日历、每日评分和 AI 建议。

项目目标不是做一个复杂的大平台，而是做一个可以每天使用、能逐步扩展、容易替换外部能力的个人工作台。

## 核心工程方针

项目采用“拼好码 / 胶水编程”方针。

核心意思是：

- 优先复用成熟能力。
- 优先使用官方 API、官方 SDK、成熟库和事实标准。
- 自研代码只负责连接、编排、适配、隔离和表达业务规则。
- 不重复造轮子。
- 不让第三方 API 的模型污染核心业务模型。
- 能配置就不要自研。
- 能适配就不要侵入。
- 能替换就不要强绑定。
- 能回滚就不要做不可逆改动。

## 协作记录规则

- 每当完成一项较大修改，需要同步更新相关文档。
- 每当完成一项较大修改，需要同步提交并推送到 GitHub。

## AI 助手相关规则

AI 助手只能生成建议，不能直接修改任务、倒计时、提醒或日历。

用户确认后，AI 助手建议才能应用。

如果后续实现“AI 助手根据描述自动确定时间”，应该让 AI 返回结构化建议，然后由用户确认创建任务、倒计时或提醒。

## 当前技术栈

- 前端：React、Vite、TypeScript、lucide-react
- 后端：NestJS、TypeScript、Prisma
- 数据库：PostgreSQL
- Redis：已接入，后续可用于提醒或后台任务
- 部署：Docker Compose
- AI：OpenAI-compatible provider
- 外部集成：Microsoft To Do，通过 Microsoft Graph
- 登录：NestJS Passport JWT、@nestjs/jwt、argon2、Prisma User 表

## 当前主要功能

### 任务管理

- 可以新建任务。
- 可以查看当前任务列表。
- 可以在列表里编辑任务。
- 可以修改任务标题。
- 可以修改任务描述。
- 可以修改任务重要性。
- 可以修改任务截止时间。
- 可以完成任务。
- 误点完成后可以恢复任务。
- 删除任务时进入归档状态，不做物理删除。
- 已完成任务只在当天显示。
- 旧的已完成任务会自动归档。
- 任务会根据截止时间和重要性自动排序。
- 修改任务内容后会重新拉取列表并自动整理排序。
- 任务列表右上角有手动重新排序按钮。
- 任务待处理数量显示在任务列表右上角。
- 任务可以显示截止倒计时。

### Microsoft To Do 同步

- 可以连接 Microsoft 账号。
- 可以从 Microsoft To Do 拉取任务到本地。
- 可以把本地任务推送到 Microsoft To Do。
- 本地完成已绑定任务时，会尝试同步完成到 Microsoft To Do。
- 本地恢复已绑定任务时，会尝试同步恢复到 Microsoft To Do。

注意：本地删除或归档任务，目前不删除 Microsoft To Do 上的任务，避免误删。

登录化后的注意事项：

- Microsoft To Do 的同步、推送接口已经要求 JWT 登录。
- 本地任务完成和恢复时，会按当前登录用户查找对应的 Microsoft To Do 绑定记录。
- OAuth callback 仍保留默认用户兼容路径；后续应该把 OAuth `state` 改成 Redis 随机 state，并绑定当前登录用户 ID。

### 登录与身份管理

- 后端已经接入 NestJS Passport JWT、`@nestjs/jwt` 和 `argon2`。
- 已新增 `POST /api/auth/register`、`POST /api/auth/login`、`GET /api/auth/me`、`POST /api/auth/logout`。
- JWT payload 使用 `{ sub: user.id, email }`。
- 前端会把 access token 存入 `localStorage`，请求 API 时自动带 `Authorization: Bearer <token>`。
- 前端收到 401 会清理登录状态并回到登录页。
- 注册时如果邮箱等于 `DEFAULT_USER_EMAIL` 且该用户仍是 `single-user-mvp` 占位密码，会给这个默认用户设置真实密码，保留原 `User.id` 和已有任务数据。
- 任务、倒计时、定时检查接口已从 `DefaultUserService` 切换为 JWT 当前用户。
- 右上角用户入口可以进入账号管理页面，当前可修改邮箱、显示名称和密码。
- 修改邮箱或账号信息后，后端会返回新的 access token，前端会刷新本地登录状态。
- `DefaultUserService` 暂时保留，主要用于 Microsoft To Do OAuth callback 的兼容路径。
- 第一版没有 refresh token、忘记密码、邮箱验证、多设备会话管理。

### 倒计时表

- 倒计时表已经从任务列表中分离。
- 倒计时用于记录“会发生的事”，不是“我要完成的事”。
- 倒计时目前复用后端的 `CalendarEvent` 模型。
- 可以新建倒计时。
- 可以查看未来倒计时列表。
- 可以显示事件还有多久发生。
- 可以设置重要性。
- 可以编辑倒计时标题。
- 可以编辑倒计时描述。
- 可以编辑倒计时重要性。
- 可以编辑倒计时发生时间。
- 可以删除倒计时。
- 倒计时表右上角有手动重新排序按钮。

### 前端布局

页面已有左侧目录菜单：

- 当前任务、倒计时、Microsoft To Do 同步归入“任务面板”。
- 定时检查、日历、提醒、每日评分、AI 助手通过左侧菜单进入独立页面。
- 左上角有按钮可以收起或展开左侧目录。

宽屏时是三列：

- A 列：新增任务、新增倒计时、Microsoft To Do 同步
- B 列：任务列表
- C 列：倒计时列表
- 新建任务面板不显示“第一阶段”字样。

手机或窄屏时，从上到下依次是：

- 任务列表
- 倒计时列表
- 新增任务
- 同步
- 新增倒计时

手机或窄屏菜单现在是侧边抽屉，不再把菜单横向展开在页面顶端。

前端 API 默认走同源 `/api`：

- Docker/Nginx 会把 `/api/` 代理到 `backend:4000/api/`。
- Vite dev server 会把 `/api` 代理到 `http://localhost:4000`。
- 因此手机访问 `http://电脑局域网IP:3000` 时，API 也会走同一个电脑地址，不再使用手机自己的 `localhost:4000`。

### 定时检查

定时检查页面已经参考 Donetick 的核心设计，改成“生活规律维护”面板，不是任务列表，也不是日程时间块。早期文档和后端底层模型仍可能使用 `RoutineHabit`、`RoutineCheckIn`、`SleepLog` 这些实现命名。

当前已完成：

- 独立数据模型：`RoutineHabit`、`RoutineCheckIn`、`SleepLog`。
- 检查事项配置，例如整理桌面、洗衣服等。
- 用户完成事项后，可以主动进入定时检查面板打卡。
- 可以跳过本轮检查事项。
- 打卡或跳过后，根据间隔和滚动规则自动计算下一次到期时间。
- 支持每 N 小时 / 天 / 周 / 月。
- 新建检查事项默认使用固定预期时间点，不随实际打卡时间漂移；例如每天 21:00 的事项，晚打卡后下一次仍按 21:00 加周期计算。
- 每条检查事项会显示预期时间、倒计时和逾期时间。
- 快要过期和已逾期的检查事项会出现在任务面板中，重要性显示为低，但数据仍保留为 `RoutineHabit`。
- 可以在检查事项列表中编辑标题、说明、分类、周期、预期时间和提醒策略。
- 可以删除检查事项。
- 最近打卡最多显示 10 条，显示区域高度约 3 条，更多内容在区域内滚动查看。
- 可以手动检查逾期检查事项，并生成本地提醒。
- 睡眠状况手动记录，包括上床时间、入睡时间、起床时间、质量和备注。

后续方向：

- 逾期检查事项可以添加到今日面板。
- 提醒自动化可接入 Redis 或后台任务。
- 后续研究自动化同步：三星手机和手环可以记录睡眠，可能可以同步到 Google，具体以后查官方能力。

实现边界：

- 定时检查不要复用 `Task` 模型。
- 逾期检测应由后端根据最后打卡时间和预设间隔计算。
- 如果加入今日面板，应该保留来源关系，不要直接把检查事项污染成普通任务。

### 电费监控

电费监控是独立模块，用来手动记录宿舍电控系统的剩余电量，并估算近期耗电速度。

当前已完成：

- 独立数据模型：`ElectricityReading`，数据库表为 `electricity_readings`。
- 左侧菜单已有“电费监控”页面。
- 可以录入读数，包括记录时间、当前剩余电量、是否充值、充值金额和备注。
- 学校系统只显示充值金额和当前剩余电量，当前按 0.63 元/度把充值金额简易换算为充值电量。
- 记录时间可手动选择，因此支持补充录入历史读数。
- 可以编辑和删除历史读数。
- 页面显示当前剩余电量、估算日耗电、预计低于 15 度的时间。
- 当前剩余电量低于 15 度时提示“电量不足”。
- 预计 3 天内低于 15 度时提示“即将不足”。

估算算法：

- 读取最近最多 30 条记录，按时间从旧到新组成相邻区间。
- 无充值区间：`耗电量 = 上次剩余电量 - 本次剩余电量`。
- 有充值且填写充值金额：先按 `充值电量 = 充值金额 / 0.63` 换算，再计算 `耗电量 = 上次剩余电量 + 充值电量 - 本次剩余电量`。
- 有充值但未填写充值金额或充值电量、时间倒序或耗电量为负的区间，不参与估算，只计入跳过区间。
- 最近最多 5 个有效区间参与加权移动平均，越近权重越高。
- 阈值固定为 15 度；后续可改成用户可配置。

实现边界：

- 电费监控不要混入 `Task` 或 `RoutineHabit` 模型。
- “定时检查”里可以单独建一个检查电费的周期事项，但电费读数仍保存在电费监控模块。
- 后续真正的通知可以复用提醒系统或浏览器通知；当前第一版只做页面内提示。

### 兴趣记录

用户提出了一个新的功能方向：类似定时检查里的打卡，但用于记录兴趣活动，例如练字、画画、阅读、乐器等。

核心区别：

- 兴趣记录只记录“我做过一次”。
- 不设置倒计时。
- 不设置下一次到期时间。
- 不产生逾期压力。
- 页面上主要提醒“上次打卡距离现在多久”，例如“上次练字是 3 天前”。
- 后续可以做频率可视化，例如最近 7 天 / 30 天次数、趋势图、热力图、间隔变化等。

实现建议：

- 不要混入 `Task` 模型。
- 不要直接复用 `RoutineHabit` 的到期和逾期逻辑。
- 可以独立建模为 `InterestActivity` 和 `InterestCheckIn`。
- 第一版只需要兴趣项目列表、打卡按钮、上次打卡时间、距离现在多久、累计次数和最近 30 天次数。
- 可视化后置，后续优先复用成熟图表库。

### 每日评分和当前任务分

每日评分后续应由多个分数加权构成，例如当前任务分、作息分、睡眠分、兴趣记录分和主观评分。

当前已先实现“当前任务分”第一版：

- 满分 100。
- 统计当前未完成且未归档的逾期任务：`status != DONE`、`status != ARCHIVED`、`dueAt < now`。
- 同时统计活跃且已逾期的定时检查事项：`isActive = true`、`state = overdue`、`nextDueAt < now`。
- 定时检查事项按 LOW 重要性参与扣分，因为它们在任务面板中也按低重要性展示。
- 没有逾期任务时为 100 分。
- 当前任务分 = `max(0, 100 - 总扣分)`。
- 单项扣分 = `重要性基础扣分 × 逾期时间系数`，单个任务最多扣 60 分。
- 重要性基础扣分：
  - LOW：4 分
  - MEDIUM：8 分
  - HIGH：14 分
- 逾期时间系数：
  - 0 到 2 小时：0.5
  - 2 到 12 小时：1.0
  - 12 到 24 小时：1.4
  - 1 到 3 天：2.0
  - 3 到 7 天：3.0
  - 7 天以上：4.0
- 任务面板标题右侧已有评分小窗，目前显示当前任务分和逾期数量；逾期数量包含普通任务和定时检查事项。
- 评分小窗未来可以扩展显示总分和其他维度分数。

## 重要业务边界

- 任务和倒计时不是同一个东西。
- 任务表示“我要完成的事”。
- 倒计时表示“会发生的事”。
- 任务可以完成、恢复、归档。
- 倒计时不能完成，只能新增、编辑和删除。
- 任务删除目前是归档。
- 倒计时删除目前是物理删除。
- Microsoft To Do 是外部能力，必须通过适配层隔离。
- Microsoft To Do 的原始数据不能直接污染本地任务模型。
- 登录后的业务接口必须从 JWT 当前用户取 userId，不能重新退回默认用户模式。
- 定时检查表示“生活规律维护”，不是普通任务，也不是倒计时或日历事件。
- 兴趣记录表示“我主动记录一次兴趣活动”，不是任务、倒计时或定时检查；它不应该有到期和逾期概念。

## 当前已知问题

Microsoft To Do 上传和同步的时间字段问题已经做了代码层修复，但还需要真实 Microsoft 账号回归验证：

- 新增 `MICROSOFT_TODO_TIME_ZONE` 配置，默认 `Asia/Shanghai`。
- 上传到 Microsoft To Do 时，把本地任务截止时间格式化为配置时区里的本地时间，并发送对应 `timeZone`。
- 从 Microsoft To Do 拉取时，按 Graph 返回的 `dateTime` 和 `timeZone` 明确还原成本地存储用的 UTC `Date`。

下一步应该用真实 Microsoft To Do 账号验证：本地创建带截止时间任务 -> 推送 -> 拉取 -> 编辑截止时间 -> 再拉取，确认时间不改变、不消失。

Docker 镜像重建时曾遇到 Docker Hub 元数据请求超时。最近一次定时检查改动已经完成本地构建、容器内 `prisma db push`，并通过标准重建启动：

```bash
docker compose up --build -d backend frontend
```
